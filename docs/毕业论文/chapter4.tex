\chapter{关键模块与平台实现}

第三章已经从需求分析和总体设计层面说明了本文平台为何采用前端展示层、后端代理层和模型推理与外部能力层组成的三层结构。本章进一步回答“系统如何落地实现”的问题。与第二章偏重技术综述、第三章偏重需求与架构不同，本章将围绕平台中的关键模块展开，说明多源检测能力是如何被组织到同一平台中的，图像与视频检测模块如何接入，本地模型与外部接口如何协同工作，以及前后端和部署层面的关键实现方式。

本章虽然进入系统实现层面，但仍以模块设计思想、关键流程和实现机制为重点，不展开为逐函数说明或代码注释式介绍。论文表达需要提炼系统特色与工程价值，因此本章主要说明仓库实现中最能体现平台能力的部分。与实验效果、性能对比和方法优劣相关的定量分析，将统一放在第五章中展开。

\section{多源检测框架实现}

\subsection{统一能力组织思路}

本文平台面临的一个核心问题是，系统所接入的能力来源并不统一。一部分能力来自本地复现模型，例如UniversalFakeDetect、Stable Diffusion和ModelScope Text-to-Video；另一部分能力来自外部平台接口，例如火山引擎视觉智能接口和方舟多模态理解接口。若让前端直接面向这些异构能力，将会面临参数格式不一致、鉴权方式不同、返回结构差异较大和异步机制复杂等问题。因此，系统在实现上首先引入了统一能力组织思路，即由后端代理层将多源能力封装为统一业务接口，再由前端按任务类型进行调用。

这一实现思路对应到代码层面，主要表现为后端服务将所有核心能力组织为检测类接口与生成类接口两大类。其中，检测类接口包括图像AI生成检测、视频AI生成检测、图像安全审核、视频安全审核以及本地图像伪造检测；生成类接口包括人脸替换、人脸动画、属性编辑、文生图、本地文生图、文生视频、本地文生视频和图生视频等。通过这样的组织方式，前端并不直接感知某个任务究竟来自本地模型还是外部平台，而只需要知道当前调用的是哪一类业务接口。

从系统实现角度看，这种统一封装的意义主要有三点。第一，它降低了前端的复杂度，使页面组件可以围绕“任务逻辑”而不是“能力来源”设计。第二，它为后续新增模型或更换接口提供了缓冲层，减少改动范围。第三，它使不同能力的结果可以在后端完成标准化处理，为后文的风险展示和实验统计提供统一数据结构。

\subsection{输入参数与结果结构标准化}

多源能力的统一接入不仅体现在接口数量的组织上，更体现在输入参数和输出结果的标准化处理上。对于输入层，平台主要围绕图像与视频两类媒体设计统一请求模式。以图像相关任务为例，后端统一接收 \texttt{imageUrl} 或 \texttt{imageBase64} 两种形式；视频相关任务则统一接收 \texttt{videoUrl} 或 \texttt{videoBase64}。这种设计直接承接了第三章提出的“文件输入与链接输入并存”的需求，同时也便于前端在不同页面之间复用上传、预览与提交逻辑。

在输出层，不同后端能力原生返回的数据结构差异较大。例如，本地检测模型更倾向于返回标签、概率与辅助信息；多模态理解接口则常常返回自然语言或嵌套结构化对象。为避免前端为每种能力单独编写渲染逻辑，系统在后端实现中对这些结果进行了归一化处理。以虚假内容检测为例，平台最终统一关注真假结论、置信度、模型名称、原因说明和辅助展示信息；以不安全内容审核为例，平台统一关注安全与否、建议处置级别、风险标签以及说明性文本。这样一来，前端页面便可以围绕统一字段完成风险标签、进度条、说明卡片和预览区域的展示。

值得注意的是，多模态大模型接口并不总是严格返回纯JSON文本，因此系统在实现中还增加了结构提取与容错逻辑。对于图像与视频理解结果，后端会先尝试从模型输出中匹配结构化对象，再执行解析和字段映射。虽然这种方式在工程上增加了一层处理，但它显著提升了结果格式的稳定性，也使大模型接口更适合被纳入一个面向用户的平台中。

\section{图像虚假内容检测模块实现}

\subsection{本地检测模型接入实现}

图像虚假内容检测模块首先接入了本地部署的UniversalFakeDetect模型，用作平台中的自主检测能力和研究基线。该模型服务通过Python接口封装后运行在独立端口，对外提供标准HTTP接口。为了让前端无需直接访问Python服务，Node.js后端对其进行了再次代理封装：前端只需访问统一的业务接口，后端再根据请求内容将 \texttt{imageUrl} 或 \texttt{imageBase64} 转换为本地模型所需的字段形式，并转发至本地推理服务。

这种二次代理的实现方式具有明显优势。一方面，它避免了前端同时管理多个服务地址的问题，使前端始终只面向后端代理层。另一方面，后端可以对本地模型返回结果进行错误检查与结构统一。例如，当本地服务不可达、返回非JSON结果或状态码异常时，后端可以将错误信息转换为平台层面可理解的提示，从而避免用户面对底层推理服务的原始报错。

从实现效果上看，本地检测模型的接入使平台具备了“可控、可复现、可离线研究”的能力。它并不依赖外部网络环境，也更适合作为论文实验中的基线方法。与此同时，本文并未将其设计为平台唯一的图像检测方案，而是将其与外部多模态检测能力并列放置，从而更符合真实应用中“多源并存”的系统构型。

\subsection{外部图像AI鉴别接口接入实现}

除本地模型外，平台还接入了基于多模态理解能力的图像AI鉴别接口。其核心实现思路是：由后端将输入图像与固定判定提示词组织为统一消息体，提交至支持视觉输入的模型接入点，再从返回结果中抽取真假判断、置信度和原因说明。与传统分类模型不同，这类接口本质上是将图像检测任务转化为“视觉输入 + 自然语言约束 + 结构化输出”的推理过程。

从工程实现角度看，图像AI鉴别接口的接入重点不在于算法本身，而在于提示词设计与结果稳定化处理。为此，后端对模型输出格式进行了严格约束，要求其仅返回包含真假字段、置信度字段和原因字段的结构化对象，并将推理温度控制在较低水平，以减少回答波动。对于模型偶尔返回自然语言解释或额外文本的情况，后端再通过正则提取和JSON解析实现结果归一化。

该模块与本地模型共同构成了图像虚假检测的双后端方案：前者强调研究可控性和自主部署，后者强调开放场景下的适应性与工程接入效率。前端页面在此基础上提供后端切换选项，使平台不仅能够完成检测任务，也能够为后续实验中不同方法的对比提供直接支撑。

\section{视频虚假内容检测模块实现}

\subsection{视频理解接口接入实现}

与图像检测相比，视频真假检测的实现复杂度更高。一方面，视频数据体量更大，上传与传输成本更高；另一方面，当前本地通用视频检测模型的复现与部署成本也明显高于图像场景。基于这一现实条件，平台当前主要采用多模态视频理解接口作为视频AI生成检测的主实现方式。具体而言，后端统一接收前端提交的视频URL或视频Base64内容，将其组织为 \texttt{video\_url} 类型输入，再与固定的视频真假判定提示词一同发送给多模态理解接口。

这一实现方案的关键不在于简单调用接口，而在于将外部能力转化为平台可用的稳定功能。为此，后端在请求构造时对视频输入格式进行了统一处理，对返回内容执行结构提取和字段映射，并根据置信度范围构造适合前端展示的结论文案。例如，在系统内部，模型原始输出会被进一步整理为是否为伪造、置信度值、模型名称、辅助说明和展示信息等字段，以便前端页面直接渲染。

从系统整体看，这种实现方式反映了本文平台“能力优先、统一封装”的工程策略。虽然当前版本尚未接入本地视频检测模型，但平台架构已经预留了对应位置，使未来若引入训练自由或专用视频检测模型，也可按与图像检测相同的方式纳入统一代理框架。

\subsection{视频检测结果结构化实现}

视频检测结果的结构化处理是该模块实现中的另一个重点。由于视频真假判别往往包含更强的不确定性，平台在结果表达上不能仅停留在简单的真假标签，而应尽量体现模型判断的置信程度和结论层级。基于这一考虑，系统在后端对视频检测输出进一步组织为“真假结论 + 置信度 + 模型名称 + 细节说明”的结构，并在前端以标签、进度条和文本解释等方式进行展示。

这种结构化表达具有两方面作用。首先，它使用户能够直观看到模型对当前视频的判断强弱，而不是被一个绝对化结论所误导。其次，它为第五章的视频实验统计提供了统一字段基础，使不同视频样本之间的对比更容易实现。因此，视频检测模块既服务于平台功能，也服务于后续实验分析与结果表达。

\section{不安全内容检测模块实现}

\subsection{图像安全审核模块实现}

图像安全审核模块的核心目标是识别图像中是否存在暴力、色情、仇恨、敏感符号或其他违规风险内容，并输出风险标签与处置建议。平台在实现上采用与图像AI鉴别相近的技术路径，即通过后端向多模态理解接口发送图像输入和安全审核提示词，再从模型回复中抽取安全结论、建议级别、标签集合和原因说明。

这一模块与虚假内容检测模块的差别主要体现在结果结构上。真假检测关注的是“是否由AI生成”，而安全审核关注的是“是否应当通过、复审或阻断”。因此，系统在后端解析结果时，不再将输出映射为真假标签，而是映射为安全状态、风险标签和处置建议，并进一步为前端展示转换为低、中、高不同风险层级。这样的实现方式使平台能够在统一UI框架下展示两类不同任务的结果，同时保持业务含义上的清晰区分。

\subsection{视频安全审核模块实现}

视频安全审核模块的实现逻辑与图像审核类似，但在输入组织和结果呈现上更强调视频语义整体性。后端统一接收视频链接或视频Base64编码，将其与安全审核提示词一起发送至支持视频理解的模型接口，并在响应后对安全结论、建议级别和标签信息进行标准化处理。前端在展示时则重点突出风险等级和建议说明，以便用户快速判断该视频是否需要人工复核。

在工程实现上，视频安全审核模块还体现了一个重要特点，即其结果不仅服务于页面展示，也为风险分级逻辑提供基础数据。审核模块与全局风险展示模块紧密配合：后端负责产出标准字段，前端负责将这些字段转化为易理解的可视化元素。这种分工方式使系统结构更清晰，也更便于后续统一调整风险分级策略。

\section{样本生成与内容管理模块实现}

\subsection{生成模块实现}

为了形成“生成—检测—分析”的闭环研究平台，系统实现了两类生成能力：一类是与Deepfake相关的人脸替换、人脸动画和属性编辑；另一类是通用场景下的文生图、文生视频和图生视频。前者主要通过火山视觉智能接口和方舟图生视频接口实现，后者则由外部生成接口与本地部署模型共同支持。

在人脸相关生成任务中，系统通过前端表单组织输入内容，并在后端完成图像编码、参数映射和接口调用。人脸替换任务重点处理双图输入与顺序匹配问题；人脸动画任务重点处理首帧图像与文本驱动的联合输入；属性编辑任务则重点支持文本指令与图像内容的组合。通用生成任务中，文生图既支持外部文生图接口，也支持本地Stable Diffusion服务；文生视频既支持外部文本到视频接口，也支持本地ModelScope Text-to-Video服务；图生视频则通过参考图与文本提示的联合输入实现动态内容生成。

这些生成模块在论文中的作用主要体现在两个方面。第一，它们使平台具备了多模态内容生产能力，可为后续检测模块构造样本。第二，它们验证了平台并不是一个孤立的“内容审查器”，而是一个可用于研究AIGC全链路安全问题的原型系统。

\subsection{数据管理模块现状与实现方式}

数据管理模块用于承接生成结果与检测记录的浏览、回看和后续使用。从当前实现情况看，平台已经具备数据管理页面骨架、内容列表展示逻辑和检测记录展示逻辑，但完整的保存、检索和导出功能仍在持续完善中。换言之，该模块在当前版本中更多承担“研究结果组织入口”的角色，而非一个完整的生产级资产管理系统。

这种现状与本文课题阶段是相匹配的。毕业设计的重点在于验证平台闭环和关键能力，而不是完全实现一个复杂的数据管理后台。因此，本文在第四章中如实说明该模块已完成的页面与接口预留工作，并将更深入的记录管理与实验数据整理放入第五章实验设计中统一讨论，这样能够保证论文叙事与项目完成度保持一致。

\section{风险分级与可视化展示实现}

平台在实现上并未将检测结果仅仅停留在接口返回层，而是进一步构建了统一的风险展示方式。对于虚假内容检测，前端主要通过真假标签、置信度进度条、模型名称和原因说明展示分析结果；对于不安全内容审核，则通过风险等级、建议级别、风险标签和说明性文本呈现审核判断。通过这种方式，不同任务虽然共享平台框架，但仍能突出各自业务含义。

在可视化实现上，平台前端采用分任务页面组织方式，将生成结果预览、检测结论和辅助说明放在同一页面中展示，以减少用户在不同页面间切换的成本。对于图像任务，页面可直接展示原图与生成图；对于视频任务，页面则以视频播放区域、文字说明和风险标签结合的方式完成展示。此外，前端还通过统一配色、标签体系和响应式布局增强页面一致性，使平台更符合教学展示和答辩演示场景。

风险分级与可视化展示模块的实现价值在于，它将底层模型推理结果转化为能够被用户快速理解的信息表达形式。对于毕业论文而言，这也是系统从“模型集合”转变为“平台系统”的关键一步。

\section{前后端平台开发与部署实现}

\subsection{前端页面与交互实现}

前端部分采用React、TypeScript、Vite和Ant Design构建。根据当前代码结构，系统已经形成首页、Deepfake生成页、通用生成页、虚假内容检测页、不安全内容检测页和数据管理页六个核心页面。路由层按业务功能进行划分，使不同页面既能独立完成任务，又保持统一导航入口。

在交互层面，前端主要围绕表单输入、文件上传、结果预览和状态提示展开。对于生成和检测任务，页面通常包含输入区、模型或任务选择区、执行按钮和结果展示区。对于结果较耗时的任务，前端还通过加载状态、提示信息和轮询等待结果等方式改善用户体验。与此同时，前端通过统一的接口访问封装，将所有请求都指向后端代理层，从而避免在页面组件中散落不同服务地址与鉴权逻辑。

平台首页与模块导航界面如图~\ref{fig:platform-home} 所示。首页并不直接承担模型推理任务，而是用于组织生成、检测和内容管理等功能入口，使用户能够从统一入口进入不同安全分析流程。这种页面组织方式也与第三章提出的“统一入口、分任务展示、结果结构化”的前端设计原则相对应。

\subsection{后端代理与任务调度实现}

后端服务基于Node.js和Express实现，是整个平台的能力汇聚中心。它在实现上主要承担四类职责：一是统一接收前端请求并按任务路由至不同能力；二是处理外部接口鉴权，包括Bearer Token和HMAC-SHA256签名两种主要模式；三是对多模态大模型返回结果进行结构提取与统一封装；四是对异步生成任务执行轮询调度，使前端能够以相对简洁的同步调用方式获得最终结果。

其中，异步任务封装是后端实现中的关键点之一。以视频生成任务为例，外部平台通常采用“提交任务—轮询查询—返回结果”的模式，如果前端直接处理这些步骤，将显著增加页面逻辑复杂度。后端通过统一封装轮询逻辑，将复杂异步流程收敛为单一业务接口，从而大幅简化前端实现。该设计直接体现了第三章中提出的“后端代理层承担差异消解与流程编排”的总体思路。

后端代理层对同步检测任务和异步生成任务的处理流程如图~\ref{fig:backend-dispatch-flow} 所示。对于图像检测等同步任务，后端主要完成参数转换、接口调用和结果归一化；对于视频生成等异步任务，后端还需要承担任务创建、轮询查询和超时处理。通过这种封装，前端能够以统一业务接口调用不同能力，系统也因此具备较好的可维护性和扩展性。

\subsection{模型服务部署实现}

本地模型服务采用独立进程方式运行，其中图像虚假检测、本地文生图和本地文生视频分别封装为独立服务，并通过不同端口提供推理接口。结合当前项目进展，这些服务与后端代理进程共同部署在实验室服务器中，由前端通过后端统一访问。平台整体采用前后端分离部署方式：前端构建后作为静态页面运行，后端代理与本地模型服务作为独立进程驻留运行。

这种部署方式的优点在于结构清晰、便于调试和扩展。本地模型服务可以按需单独重启和替换，后端代理则负责维持稳定对外接口。对于毕业设计答辩而言，这种部署结构也更适合进行现场演示，因为前端、后端和模型服务之间的关系较为明确，出现问题时也更容易定位。尽管当前系统仍然存在视频模型资源开销较大、数据管理能力尚待完善等问题，但从整体上看，该部署方案已经能够支撑平台的核心功能运行。

\section{本章小结}

本章在第三章总体设计的基础上，进一步说明了平台关键模块的具体实现方式。首先，围绕多源检测框架，介绍了系统如何通过后端代理实现本地模型与外部接口的统一接入，以及如何完成输入参数和结果结构的标准化处理。随后，分别说明了图像虚假内容检测、视频虚假内容检测、不安全内容审核、样本生成与结果管理等模块的实现逻辑，并分析了风险分级与可视化展示在平台中的作用。

在工程实现层面，本章还总结了前端页面组织、后端代理与任务调度以及本地模型服务部署的主要做法，体现了平台从需求分析到具体落地的完整链路。可以看到，本文系统的特点并不在于某一个单独模型本身，而在于通过统一组织与模块化实现，使多类能力在同一平台中协同工作。下一章将在本章实现基础上，围绕实验环境、样本构建、检测效果与系统性能开展定量分析，以验证该平台方案的有效性与实用性。
